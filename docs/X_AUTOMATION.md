# X自動投稿システム 運用仕様

## 1. 概要

世界経済天気予報の既存データを使い、日本時間の07:15、12:10、21:30に投稿候補を生成します。初期状態は`DRY_RUN=true`、`AUTO_POST_ENABLED=false`です。条件を満たす出典がない日は投稿を作らず、停止理由を履歴へ保存します。

現在の推奨運用は半自動です。管理画面で投稿候補を生成・確認し、「投稿文をコピー」または「Xで確認して投稿」から人が最終公開します。この方法はX投稿APIを呼ばないため、X APIクレジットを消費しません。

## 2. 処理フロー

1. 既存のFRED/米財務省/NY Fedデータ、Yahoo Chart、GDELT、公的RSS、SEC 13Fを収集
2. 固定しきい値と20日z-scoreで市場異常を検出
3. 市場異常、投資家情報、経済ニュース、話題量から最大10候補を生成
4. 重要度、鮮度、話題量、日本関連性、波及性、意外性、先行き関心度を採点
5. 上位3テーマを材料にライターAIが文体の異なる3案を生成
6. 各案を別リクエストのファクトチェックAIで検証
7. 編集長AIが合格案だけから1案を選定
8. 禁止表現、X加重文字数、直近投稿との類似度、URL回数をコードで再検証
9. DRY_RUNなら保存のみ。本番モードならX APIへ投稿
10. 本文、候補、出典、各スコア、投稿ID、エラーを永続化

## 3. データソース

- 信用・金利・雇用・流動性: 既存FRED、NY Fed、米財務省実装
- 株価指数・為替・商品・暗号資産: Yahoo Chart API
- ニュース: GDELT、Federal Reserve/FDIC/SEC公的RSS。著名投資家・主要中央銀行関係者は専用人物クエリを使い、Reuters、Bloomberg、FT、日経、公式ドメイン等に限定
- 著名投資家ポジション: SEC 13F

13Fは最大45日遅れます。発言情報として扱わず、SEC提出ポジションの補助シグナルとして扱います。公式一次情報と別ソースの確認が揃わない人物情報は投稿候補から外れます。

## 4. 時刻とスケジューラー

スケジュールはUTCで定義します。

| 枠 | 日本時間 | UTC cron |
|---|---:|---:|
| morning | 07:15 | `15 22 * * *`（前日UTC） |
| midday | 12:10 | `10 3 * * *` |
| evening | 21:30 | `30 12 * * *` |

Cronは`Authorization: Bearer $CRON_SECRET`で保護します。実行時刻から45分以上離れている場合は投稿を停止します。

Vercel HobbyはCronが1日1回までのため、本プロジェクトでは`.github/workflows/x-post-schedule.yml`を使用します。GitHub Actionsが3回/日、`CRON_SECRET`で保護したVercel APIを呼びます。投稿処理そのものはVercel Functions上で実行されます。

## 5. 永続化

- ローカル: `data/x-automation/state.json`（Git対象外）
- Vercel: Upstash Redis REST

Vercelの実行環境ではローカルファイルを永続化できないため、Upstash未設定なら処理を失敗させます。前回履歴を捨てて二重投稿するフォールバックは行いません。

## 6. 管理画面

`/admin/x-automation`へアクセスし、`ADMIN_SECRET`でログインします。

- 次回投稿予定
- DRY_RUN/自動投稿/URL誘導/2情報源必須の切り替え
- 異常値しきい値
- 朝・昼・夜枠の手動実行
- 候補、出典、バイラルスコア、リスクスコア
- 選定本文の手動修正
- 投稿履歴、投稿ID、エラー履歴
- X権限で取得できる公開表示回数・いいね・リポストの手動同期
- 必要環境変数の設定状況

管理CookieはHttpOnly、SameSite Strict、12時間有効です。

## 7. DRY_RUN確認手順

1. `.env.local`へFRED、LLM、管理用の環境変数を設定
2. `DRY_RUN=true`と`AUTO_POST_ENABLED=false`を確認
3. `npm run dev`
4. `/admin/x-automation`へログイン
5. 任意の投稿枠で「生成」を実行
6. 候補、出典、ファクトチェック結果、最終文、エラーを確認
7. 最低1日、3枠のCronまたは手動実行を確認

## 8. 本番投稿の有効化

1. X Developer Consoleで対象アプリに投稿権限を付与
2. X OAuth 1.0a User Contextの4資格情報をVercelへ登録
3. LLM、Upstash、`CRON_SECRET`、`ADMIN_SECRET`を登録
4. DRY_RUNで3枠を確認
5. `DRY_RUN=false`、`AUTO_POST_ENABLED=true`へ変更
6. 管理画面でもDRY_RUNをOFF、自動投稿をONにする
7. 管理画面で環境表示がすべて有効か確認

環境変数と管理画面の二重ゲートを両方有効にしない限りXへは投稿しません。

## 9. 停止条件

- 2つの独立した情報源を確認できない
- 値または公開日時を確認できない
- 48時間を超えた一般ニュースしかない
- 全案がファクトチェック不合格
- X加重文字数が280超
- 禁止表現を含む
- 直近投稿との類似度が高い
- 同じ投稿枠を処理済み
- Cronが45分超遅延
- X認証または永続ストレージが未設定

## 10. 既知の制限

- Yahoo Chartは非公式インターフェースでSLAがありません。失敗時は他ソースを推測補完しません。
- GDELTは記事発見に使い、見出しだけで原因を断定しません。
- 13Fはリアルタイムの発言・売買情報ではありません。
- Xの表示回数等は権限とプランによって取得可否が異なります。管理画面の同期に失敗しても投稿履歴は維持します。
- Google Trends/X言及量は公式に安定取得できる資格情報がないため、初期版の話題量はニュース件数の増加で代替しています。
